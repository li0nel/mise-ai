5. Responsive Notes

  • This mock is mobile-native (390px phone shell). No desktop or tablet adaptation needed for the in-app experience.
  • Carousel cards: Center card width is 296px (76% of 390px shell width), leaving exactly enough room for peek cards at the edges.
  • Swipe gesture: In production, use react-native-reanimated + react-native-gesture-handler for spring-physics horizontal pan. Snap points at each card index. Velocity-based flick detection (threshold ~500px/s).
  • Card scaling: Cards scale from 0.90 to 1.0 as they move toward center using an interpolated animated value. Opacity interpolates from 0.45 to 1.0.
  • Dot indicators: Active dot width animates from 8px to 24px using Animated.spring with damping: 15, stiffness: 150.
  • Mock page layout: The frame gallery wraps on screens narrower than ~860px (two phones side-by-side). Below 420px, frames stack vertically.

6. What the Mock Does NOT Show

  • Swipe animation: The actual horizontal pan gesture with spring physics, snap-to-card behavior, and velocity-based flick detection. Implementation requires PanGestureHandler and useAnimatedStyle.
  • Card enter/exit transitions: When advancing to the next card, the current card should slide left and scale down while the next card slides in from the right and scales up. Duration: 300ms, easing: cubic-bezier(0.34, 1.56, 0.64, 1) (spring overshoot).
  • Checkmark animation: The overlay should fade in (200ms) and the checkmark should draw itself with an SVG stroke-dasharray animation (400ms, ease-out). Total: 600ms staggered.
  • Haptic feedback: Light impact on card snap, medium impact on confirm, success notification on all-answered.
  • Loading state: If question generation takes time, show a skeleton card (pulsing gradient) in the center slot with "Preparing your questions..." label.
  • Error state: If AI fails to generate questions, show an error card: "Something went wrong" with a "Try Again" button in the center slot.
  • Network failure: If offline, show cached questions if available, otherwise an offline illustration with "Connect to continue."
  • Keyboard navigation: Left/Right arrow keys should swipe between cards. Enter/Space should toggle pill selection. Tab should cycle through pills within the active card.
  • Screen reader announcements: On card change: "Question [N] of [total]: [question text]". On answer: "[option] selected". On all answered: "All questions answered. Your recipe is ready."
  • prefers-reduced-motion: Disable swipe spring animations, use instant card transitions. Disable celebration bounce. Keep opacity transitions.
  • Multi-select vs single-select: Some questions (dietary, sides) allow multi-select. Others (protein, spice level) are single-select. The pill behavior must differ: single-select deselects siblings, multi-select toggles independently.
  • Dynamic question count: The number of questions (and therefore dots) varies by recipe. The carousel must handle 3-8 cards. With 8 cards, dots should shrink to 6px to fit.
  • Recipe banner slide-up timing: Banner appears 600ms after the last checkmark overlay completes, with a spring slide-up animation (400ms, cubic-bezier overshoot).
  • Transition to celebration: 800ms after banner appears, carousel scales down (0.92) and blurs (2px) over 400ms. Celebration content fades in (300ms) immediately after.
